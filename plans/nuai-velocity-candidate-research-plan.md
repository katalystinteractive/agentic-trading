# NUAI-Like Velocity Candidate Research

Status: Draft for implementation planning  
Scope: Build a separate advisory scanner/report for high-cycle, recent-performance tickers like NUAI.  
Non-goal: Do not change current tournament findings, bullet recommendations, deployment decisions, or portfolio state.

## Verified Analysis

The current recency-weighted composite scoring improves existing sweep and tournament inputs, but it does not produce a dedicated list of NUAI-like tickers to use. The scoring change affects tools that call `multi_period_scorer.compute_composite()`, while the user needs a separate discovery output that mines already-collected artifacts for a different profile: frequent, recent, repeatable 5-7 percent cycling where capital turns over quickly.

Existing `tools/velocity_scanner.py` and `tools/velocity_dashboard.py` do not satisfy this need. `velocity_scanner.py` downloads current Yahoo data for one ticker and scores a live technical signal. `velocity_dashboard.py` scans tickers already listed in `portfolio.json["velocity_watchlist"]`. It does not discover candidates from support/resistance/bounce/regime-exit sweep artifacts or cycle timing files.

The new capability must be separate and advisory. It may read existing artifacts and may write a new report, but it must not write to `portfolio.json`, `data/tournament_results.json`, current bullet recommendation outputs, or any live mutation path. Later reporting integration into `status_gatherer.py` or `morning_assembler.py` is acceptable only as a clearly labeled advisory section, because those are reporting surfaces rather than ranking or portfolio mutation paths.

The local artifacts contain enough evidence for an initial candidate list, but their schemas are not uniform:

- `data/support_sweep_results.json` may contain `stats` / `periods` from support stages and `slippage_stats` / `slippage_periods` from slippage stage output.
- `data/resistance_sweep_results.json` uses `stats` / `periods`.
- `data/bounce_sweep_results.json` uses `stats` / `periods`.
- `data/regime_exit_sweep_results.json` uses `regime_exit_stats` / `regime_exit_periods`.
- `data/sweep_support_levels.json` is a legacy/support-level sweep artifact that also uses `stats` / `periods` and currently contains useful NUAI-like evidence.
- `tickers/<TICKER>/cycle_timing.json` stores cadence under `statistics`, including `total_cycles`, `median_first`, `median_deep`, and `immediate_fill_pct`.

One current artifact problem must be preserved in the plan: `data/entry_sweep_results.json` is malformed at the time of analysis (`Extra data: line 46 column 2`). The first implementation must exclude entry sweep data by default or quarantine it as invalid, rather than failing the whole scanner or consuming partial data.

## Goal

Create a separate research tool that outputs a ranked advisory list of NUAI-like velocity candidates from already-downloaded pipeline artifacts.

The list should answer:

- Which tickers look like NUAI based on recent, repeated, profitable cycling?
- Which candidates are already tracked, in tournament rankings, or completely untracked?
- Which candidates have enough recent evidence to deserve manual review for a parallel velocity strategy?
- Which candidates are disqualified or low confidence because their edge is stale, only 12-month driven, missing cycle timing, or based on malformed data?

Implement a normalizer that converts each per-ticker, per-strategy artifact entry into:

```python
{
    "ticker": "VCAND",
    "strategy": "support",
    "source_file": "data/support_sweep_results.json",
    "stats": {
        "composite": 0.0,
        "pnl": 0.0,
        "trades": 0,
        "win_rate": 0.0
    },
    "periods": {
        "12": {"pnl": 0.0, "cycles": 0, "trades": 0, "win_rate": 0.0},
        "6": {"pnl": 0.0, "cycles": 0, "trades": 0, "win_rate": 0.0},
        "3": {"pnl": 0.0, "cycles": 0, "trades": 0, "win_rate": 0.0},
        "1": {"pnl": 0.0, "cycles": 0, "trades": 0, "win_rate": 0.0}
    },
    "winner_context": {}
}
```

Normalization rules:

- For support entries, prefer `slippage_stats` / `slippage_periods` when present because current `data/support_sweep_results.json` may be the slippage-stage output. Fall back to `stats` / `periods`.
- For legacy/support-level entries from `data/sweep_support_levels.json`, use `stats` / `periods` and set `strategy` to `support_levels`.
- For resistance entries, use `stats` / `periods` and preserve `vs_flat` in `winner_context`.
- For bounce entries, use `stats` / `periods` and preserve `vs_others` in `winner_context`.
- For regime exit entries, use `regime_exit_stats` / `regime_exit_periods`.
- Ignore metadata keys beginning with `_`.
- Coerce missing or malformed numeric fields to zero and add a per-ticker warning.

## Cycle Amplitude Semantics

"5-7 percent cycling" means the ticker has enough repeated support-to-resistance swing to support a 5-7 percent profit target, not that every full cycle must be capped at exactly 5-7 percent.

Use `cycle_timing.json` cycles to estimate tradable swing amplitude when available:

```python
first_rebound_pct = (resistance_price - first_touch_price) / first_touch_price * 100
deep_rebound_pct = (resistance_price - deep_touch_price) / deep_touch_price * 100
```

Use the median of valid `first_rebound_pct` values as `median_cycle_rebound_pct`. If `first_touch_price` is missing but `deep_touch_price` is present, use the deep rebound value for that cycle. Ignore cycles with missing, zero, or negative prices.

Amplitude gates:

- Strong amplitude: `median_cycle_rebound_pct >= 7`.
- Acceptable amplitude: `median_cycle_rebound_pct >= 5`.
- Missing amplitude: allowed, but confidence cannot be `high` unless either `1m trades >= 4`, `1m win_rate >= 85`, and `1m pnl > 0`, or `3m trades >= 10`, `3m win_rate >= 85`, and `3m pnl > 0`.
- Low amplitude: `median_cycle_rebound_pct < 5`; classify as `low_amplitude_review` unless either `1m trades >= 6`, `1m win_rate >= 90`, and `1m pnl > 0`, or `3m trades >= 15`, `3m win_rate >= 90`, and `3m pnl > 0`.

## Candidate Profile Definition

A NUAI-like velocity candidate should be scored using recent evidence first.

Required hard gates for default inclusion:

- At least one normalized strategy has `3m trades >= 4` or `1m trades >= 2`.
- Recent win rate is strong: `1m win_rate >= 75` when `1m trades >= 2`, or `3m win_rate >= 75` when `3m trades >= 4`.
- Recent PnL is positive in the same period used for the win-rate gate.
- Ticker is not marked as winding down in `portfolio.json["positions"]`.

Soft gates that add confidence:

- `cycle_timing.statistics.total_cycles >= 5`.
- `cycle_timing.statistics.median_first <= 2`.
- `cycle_timing.statistics.immediate_fill_pct >= 80`.
- `median_cycle_rebound_pct >= 7`, or at least `>= 5` with strong 1m/3m sweep evidence.
- Tournament rank exists but is outside top 30, which identifies candidates the current tournament underweights.
- Portfolio/tournament status is context only. It should not add points by itself. Report statuses such as `position`, `watchlist`, `velocity_watchlist`, `candidate`, `untracked`, and `winding_down_excluded` so the operator can decide what is actionable.

Staleness penalty:

- A ticker with strong 12m metrics but no 1m or 3m activity should be classified as `stale_history`, not ranked as a primary candidate.
- 12m data should only contribute stability context, not the main score.

## Scoring Model

Use a simple explainable score from 0 to 100.

Recommended starting weights:

- 35 points: recent profitability and win rate.
- 25 points: cycle cadence from `cycle_timing.json`.
- 15 points: trade frequency, with 1m and 3m trades favored.
- 10 points: cycle amplitude, using `median_cycle_rebound_pct`.
- 10 points: multi-strategy confirmation across support, resistance, bounce, and regime-exit.
- 10 points: tournament underweight opportunity, meaning rank outside top 30 or missing from tournament despite strong recent metrics.

Normalize the final score back to 0-100 after component calculation. Do not give points merely for being tracked or untracked; that is output context, not ranking evidence.

Scoring must expose component details in JSON and markdown so the operator can see why a ticker ranked high.

## Markdown Report Contract

Write `data/velocity_candidate_research.md` with:

- Title and generation timestamp.
- Advisory-only warning.
- Summary counts:
  - total candidates scanned
  - candidates passing gates
  - high-confidence candidates
  - tracked candidates
  - untracked candidates
  - malformed or skipped inputs
- Top candidates table:
  - rank
  - ticker
  - score
  - confidence
  - status
  - tournament rank
  - best strategy
  - 1m trades / win / PnL
  - 3m trades / win / PnL
  - median first touch
  - immediate fill percent
  - median cycle rebound percent
  - why it matched
- Separate sections:
  - `High Confidence`
  - `Needs Manual Review`
  - `Stale 12m Only`
  - `Skipped or Invalid Inputs`

Markdown sections should map directly to JSON buckets:

- `High Confidence` comes from `rankings` entries with `confidence == "high"`.
- `Needs Manual Review` comes from `needs_manual_review`.
- `Stale 12m Only` comes from `stale_history`.
- `Skipped or Invalid Inputs` comes from `skipped_inputs`.
- `excluded` remains in JSON and may be summarized in markdown counts, but should not be mixed into candidate recommendation tables.

The markdown must avoid commands like "buy now" or "add to watchlist". Use advisory language such as "candidate for review".

## Acceptance Criteria

- Running `python3 tools/velocity_candidate_research.py` produces a separate ranked advisory report.
- The report identifies NUAI-like candidates using recent 1m/3m evidence first.
- The report includes cycle-amplitude evidence or a missing-amplitude confidence penalty.
- 12m performance cannot be the main reason a ticker ranks high.
- Existing current findings and recommendations are unaffected.
- No current live artifact, tournament output, bullet recommendation, deployment recommendation, or portfolio state is modified.
- Malformed entry sweep data does not crash the tool.
- Tests cover all supported input schema variants and the separation contract.
- JSON and markdown category buckets are consistent with each other.
